The Requirements

Our Client for this project is Dr. Atia Rafiq on behalf od NHS Digital. We have also been working closely with Hiroki Inagaki from NTT Data who provided us with the technical help that we needed. We had weekly meetings with Hiroki in order to ensure that our project was on schedule.

We summarised the initial problem given by our client Dr Atia Rafiq as follows:
“Provide medical students with a learning tool that will help them follow a GP consultation whilst not disrupting its general flow. “

The technology we have used is a pair of smart glasses from Vuzix and an Android smartphone. The user will have the glasses on during the consultation. As medical words are said, they will appear on the glasses and the user will be asked whether or not they wish to show a short definition of the term. The user can then show the definition for the current word or display the next word by using 2 easily accessible buttons on the phone application.

The general problems that we considered through our design iterations are: Managing the user interaction so that it does not disrupt the consultation, displaying the words for a correct amount of time so that the screen does not clutter and showing the definition long enough for the user to not lose focus.

From this initial meeting, we set up a list of requirements that we refined throughout the development of the project to suit the clients' needs as best as possible. The project goals are listed below, followed by a detailed list of our final requirements.

Project Goals

  • Design an application that is simple and easy to use
  • Find a solution that does not disrupt the users's concentration
  • Provide some functionality for the user to come back to words for which they displayed a definition
  • Make the project expandandable to other fields and use cases
  • Allow flexible usage of the application (On the phone only or with the glasses

MoSCoW [1] Style Requirements

Once we defined our project goals, we wrote up a list of project requirements that we confirmed with our clients.

The solution must:
  • Implement voice recognition to understand the appropriate words (diseases and medicines)
  • Have a way for the user to accept or decline showing a definition of a word
  • Query a medical database in order to get the appropriate definitions
  • Display a short definition to the user in a simple manner
  • Be discrete and subtle so that it doesn’t interrupt the consultation
The solution should:
  • Keep a log of the words that the user showed the definitions of
  • Allow the user to access these words after the consultation through a mobile interface
  • Display definitions for each word on the mobile interface
The solution could:
  • Learn what words the user knows and not display them

Personas

Establishing our 2 personas played an important role in deciding how to implement the features to suit our requirements. We wanted to make the application as easy to use as possible by as many people as possible. Our 2 Personas are both medical students, our target user audience for "Smarter Medics". We took into account several factors:

- Familiarity with technology: We centred our development around simplicity and ease of use
- Social Behaviour: We provide 2 usage options to suit different situations, using the phone only or using the glasses too.
- Technical knowledge: We added a blacklist of terms that won't come up
