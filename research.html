Our Research

In this section, we will be describing the research that we conducted at various stages of development. We will talk about our research on devices, external dictionary services and speech recognition APIs. In each section we will see the options we considered along with an explanation detailing our choice for the final option. For research relating to Human Computer Interaction, please visit our HCI section. Given that our client wanted a smart glasses application, the technologies available are limited, hence why our research is concise. In our minds, we have opted for the optimal choices in terms of feasability.

Devices

For our project we had to chose 2 devices: a phone and a pair of smart glasses. We had 2 options for our glasses: The google glasses or the Vuzix M-100 [1]. Below is a table comparing both devices:

Feature Vuzix M-100 Google Glasses
Price $1000 $1500
CPU OMAP4460 at 1.2GHz OMAP 4430 SoC, dual-core @ 1GHz
Memory 1 GB RAM 1GB RAM to 2GB RAM
Control 4 control buttons, remote control app, voice navigation and gestures. Touchpad, app and voice
Bluetooth Yes Yes
Sound Ear speaker and a noise canceling microphone. Bone conduction transducer, ear buds, and microphone.

The comparison shows us that the Google Glasses have higher technical specifications but we opted for the Vuzix M-100 for a few reasons:

  • More affordable
  • Our clients from NTT Data had valuable expertise with them
  • Better controls available, making it easier for the user to navigate
  • Ease of deploying applications since it runs android natively and doesn't require uploading a different application
For our phone, the choice between operating systems was simple; given the fact that the glasses are running android, we opted for the use of an android device. A plus side of this is that android smartphones are available for much cheaper than iOS devices and hence reduce the cost of using our solution. In the future, teams mmight look to implement an iOS app but we have left it out for our project for timing reasons.

Programming languages

Given that we were going to use an Android phone and glasses running Android natively, we have programmed our application in Java using Android Studio. We have used the Android documentation [2] extensively in order to implement the app's features. Given that we were familiar with java, this let us focus on development rather than learning a new language frokm scratch.
For data storage, we had a choice between json and XML as they are the 2 most popular serialisation languages available. We opted with XML because as described below, our dictionary API sued the same language. We could then reuse our knowledge and code for writing to XMl documents and parsing them. We had initially thought of having an online database to store the words, however we went with local storage in XML as this makes the application faster, and will be useful in the future if someone would like to collect data about our application.

Voice recognition and dictionary services

For voice recognition, we considered various options including using google's Cloud API [3], as well as third party services. However, we settled on using the in-build speech to text API on Android phones. We decided to go with this for our solution as it has no cost of usage unlinke google cloud and will also ensure better data privacy since the conversations are not sent to external sources unlinke with third party APIs. Additionally, given that we have programmed our application in java using Android libraries, this solution was easier to implement.

With the dictionary services, we initially had thought of using Text Mining algorithms in order to search the output text for medical terms and then pass them onto a medical Database. However, this solution would have taken us too long to implement in the given time frame and can always be added in the future. Instead, we have used Meriam Webster's medical dicitonary online API [4]. It is a REST API that allows us to query the server for a word and that returns detailed information relating to dictionary attributes. We chose this as our solution because word definitions would be more accurate and we can just check if a word is contained inside the medical dictionary. Compared to text mining, this alternative saves us time as there is no required algorithm, but it depends on the server being up. If we were to carry this project out further, I would suggest changing the way of retrieving medical definitions as there is a limit to the volume of API calls given that the service is free to use.

Summary

For our final solution, we opted with using an Android phone alongside the Vuzix M-100 Smart Glasses. We have used Android's in-build Speech to Text functionality and Meriam Webster's online medical dictionary API for picking up words and getting their definitions.

The decisions we made took into account the time we had to complete the project as well as our individual expertise. In the end we managed to deliver a working solution, a product of our good choices.
